# kubectl Commands & YAML Manifests Interview Questions

## Most Frequently Asked Interview Questions

### Question 1

**Question**

What is `kubectl`, and why is it important in Kubernetes?

**Answer**

`kubectl` is the official Kubernetes command-line tool used to communicate with the Kubernetes API Server. It allows administrators and developers to create, manage, update, troubleshoot, and delete Kubernetes resources.

**Explanation**

Every `kubectl` command sends requests to the API Server, which validates the request and updates the cluster state stored in etcd.

**Used in Production**

- Deploying applications
- Managing Pods and Deployments
- Troubleshooting cluster issues
- Monitoring resources

**Common Mistake**

Many candidates think `kubectl` communicates directly with worker nodes. In reality, it communicates with the API Server.

---

### Question 2

**Question**

What is the difference between `kubectl create` and `kubectl apply`?

**Answer**

- `kubectl create` creates a resource only if it does not already exist.
- `kubectl apply` creates the resource if it doesn't exist or updates it if it already exists.

**Explanation**

`kubectl apply` is preferred for Infrastructure as Code because it supports declarative resource management.

**Used in Production**

CI/CD pipelines almost always use `kubectl apply`.

**Common Mistake**

Using `kubectl create` repeatedly results in "resource already exists" errors.

---

### Question 3

**Question**

What does the `kubectl get` command do?

**Answer**

`kubectl get` lists Kubernetes resources.

Examples:

```bash
kubectl get pods
kubectl get deployments
kubectl get services
kubectl get nodes
```

**Explanation**

It provides a quick overview of resource status within the cluster.

**Used in Production**

Checking running workloads and verifying deployments.

---

### Question 4

**Question**

When should you use `kubectl describe` instead of `kubectl get`?

**Answer**

`kubectl describe` provides detailed information about a resource, including:

- Events
- Conditions
- Labels
- Resource configuration
- Recent errors

Example:

```bash
kubectl describe pod nginx
```

**Explanation**

It is primarily used for troubleshooting.

**Used in Production**

Investigating failed Pods and scheduling issues.

---

### Question 5

**Question**

What is the purpose of the `kubectl logs` command?

**Answer**

`kubectl logs` displays the logs generated by a container.

Example:

```bash
kubectl logs nginx-pod
```

**Explanation**

Logs help diagnose application failures and runtime issues.

**Used in Production**

Application troubleshooting and debugging.

**Common Mistake**

Using `kubectl describe` to view application logs. Application logs are viewed with `kubectl logs`.

---

### Question 6

**Question**

What does the `kubectl exec` command do?

**Answer**

`kubectl exec` executes commands inside a running container.

Example:

```bash
kubectl exec -it nginx-pod -- /bin/bash
```

**Explanation**

It is commonly used for debugging, inspecting files, and verifying application behavior.

**Used in Production**

- Checking configuration files
- Verifying installed software
- Running diagnostic commands

---

### Question 7

**Question**

How does the `kubectl scale` command work?

**Answer**

`kubectl scale` changes the number of Pod replicas managed by a Deployment, ReplicaSet, or StatefulSet.

Example:

```bash
kubectl scale deployment nginx --replicas=5
```

**Explanation**

The Deployment Controller creates or removes Pods to reach the desired replica count.

**Used in Production**

Handling increased or decreased application load.

---

### Question 8

**Question**

What is the purpose of the `kubectl rollout` command?

**Answer**

`kubectl rollout` manages Deployment rollouts.

Common commands:

```bash
kubectl rollout status deployment nginx

kubectl rollout history deployment nginx

kubectl rollout undo deployment nginx
```

**Explanation**

It helps monitor deployments and perform rollbacks.

**Used in Production**

Managing application releases.

---

### Question 9

**Question**

What is a Kubernetes YAML Manifest?

**Answer**

A YAML Manifest is a configuration file that declaratively defines Kubernetes resources.

Example resources include:

- Pod
- Deployment
- Service
- ConfigMap
- Secret

**Explanation**

Kubernetes reads the YAML file and creates the desired resources.

**Used in Production**

All Infrastructure as Code deployments.

---

### Question 10

**Question**

What are the main sections of a Kubernetes YAML Manifest?

**Answer**

The primary sections are:

- `apiVersion`
- `kind`
- `metadata`
- `spec`

**Explanation**

- **apiVersion** specifies the Kubernetes API version.
- **kind** defines the resource type.
- **metadata** contains resource information such as name and labels.
- **spec** defines the desired configuration.

**Used in Production**

Every Kubernetes manifest includes these sections.

---

### Question 11

**Question**

What is the purpose of the `metadata` section in a Kubernetes manifest?

**Answer**

The `metadata` section identifies the resource.

It typically contains:

- Name
- Namespace
- Labels
- Annotations

**Explanation**

Metadata uniquely identifies resources and enables selection and organization.

**Used in Production**

Application grouping and resource management.

---

### Question 12

**Question**

What does the `spec` section define in a Kubernetes manifest?

**Answer**

The `spec` section describes the desired state of the resource.

For example, in a Deployment it defines:

- Number of replicas
- Container image
- Ports
- Environment variables
- Resource limits

**Explanation**

Kubernetes continuously works to ensure the actual cluster state matches the desired state described in the `spec`.

**Common Mistake**

Confusing `metadata` (resource identity) with `spec` (resource configuration).

# Scenario-Based Interview Questions

### Scenario 1

**Question**

A Pod is stuck in the Pending state. Which commands would you use to troubleshoot the issue?

**Answer**

```bash
kubectl get pods

kubectl describe pod <pod-name>

kubectl get nodes
```

**Explanation**

`kubectl describe` provides scheduling events that identify why the Pod cannot be scheduled.

---

### Scenario 2

**Question**

A Deployment has been updated, but users report application failures. How can you verify the rollout status?

**Answer**

```bash
kubectl rollout status deployment <deployment-name>
```

If necessary:

```bash
kubectl rollout undo deployment <deployment-name>
```

**Explanation**

The rollout command monitors deployment progress and supports quick recovery.

---

### Scenario 3

**Question**

Your application is crashing immediately after startup. Which command would you use first?

**Answer**

```bash
kubectl logs <pod-name>
```

**Explanation**

Application logs usually reveal startup errors such as configuration issues or missing dependencies.

---

### Scenario 4

**Question**

A developer needs temporary access inside a running container for debugging. Which command should they use?

**Answer**

```bash
kubectl exec -it <pod-name> -- /bin/bash
```

**Explanation**

`kubectl exec` opens an interactive shell inside the container for troubleshooting.

---

### Scenario 5

**Question**

Traffic to your application has doubled. You need to increase the number of running Pods immediately. Which command should you use?

**Answer**

```bash
kubectl scale deployment web-app --replicas=6
```

**Explanation**

Scaling increases the number of Pod replicas without redeploying the application.

---

### Scenario 6

**Question**

Your CI/CD pipeline deploys applications using YAML files. Which command should it use to ensure resources are created or updated?

**Answer**

```bash
kubectl apply -f deployment.yaml
```

**Explanation**

`kubectl apply` supports declarative resource management and safely updates existing resources.

---

### Scenario 7

**Question**

You accidentally created an incorrect Service in production. Which command should you use to remove it?

**Answer**

```bash
kubectl delete service <service-name>
```

**Explanation**

The `kubectl delete` command removes Kubernetes resources from the cluster.

---

### Scenario 8

**Question**

A developer modified a Deployment directly in the cluster to fix an urgent issue. Which command allows editing a resource without modifying the YAML file locally?

**Answer**

```bash
kubectl edit deployment <deployment-name>
```

**Explanation**

`kubectl edit` opens the resource in the default editor, allowing immediate updates.

---

### Scenario 9

**Question**

Your Deployment YAML specifies an incorrect container image. Which section of the manifest should you modify?

**Answer**

Update the image under the `spec` section.

**Explanation**

The `spec` defines the desired configuration of the Deployment, including container images.

---

### Scenario 10

**Question**

You need to identify which Kubernetes API version should be used when creating a Deployment manifest. Which field specifies this information?

**Answer**

The `apiVersion` field.

Example:

```yaml
apiVersion: apps/v1
```

**Explanation**

The API version tells Kubernetes which API should be used to interpret the resource definition.
